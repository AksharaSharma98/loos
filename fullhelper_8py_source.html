6 fullhelper_fullhelp_message = """
7 This tool helps you wrap fullhelp messages for loos C++ source so that
8 they will be formatted correctly in the WYSIWYG loos style, where all the
9 needed punctuation and whitespace are specified in source. The objective was to
10 provide a way for users to write their desired fullhelp message in softwrapping
11 text editors, fairly common these days, as a plain text file. The correct
12 wrapping and line-decorating are then done by this script, and the adjusted
13 text is then written into the source at a spot specified by a locator or signal
14 string. This is done by assembling the entire updated source code as a list,
15 writing a backup of the original source code (in case something bad happens),
16 then overwriting the source file with the newly assembled list of lines of
17 source code, which SHOULD be the same in every way except the signal string
18 will be replaced by the correctly formatted lines of strings that comprise the
19 fullhelp message.
20 
21 The intention of this script is for you to write your source code as you would,
22 including a placeholder string instead of your fullhelpstring in the definition
23 of 'string fullHelpMessage(void)' such as:
24 string fullHelpMessage(void) {
25  string msg =
26  "XXX";
27 
28  return(msg);
29 }
30 
31 In this way you can continue to work on your code, able to build it and even
32 test the appearance of the options framework help phrases, without having to
33 come up with and correctly format a handsome and useful full help until the
34 end. Obviously you can use any string you want, but make good choices. This
35 script will find all instances of the signal string and replace them. This
36 means you need to include its surrounding quotes (easily done in bash by
37 including the signal string inside of doublequotes nested in single quotes, see
38 usage string). It also means that if you use a signal string you include
39 elsewhere, you will get additional copies of your fullhelp message inserted in
40 those places. This is one reason the backup gets written. The other being that
41 having the backup makes it easier to go back to editing your full help message
42 if you decide it needs to be changed after it's been inserted.
43 
44 The three obligatory arguments are the name of .txt containing your full help
45 message, the source code file you'd like to sub your message into, and the
46 locator string you are using as a target for replacement by the full help
47 message. The fourth, optional, argument is the column count to wrap to. The
48 default column count is 80.
49 """
