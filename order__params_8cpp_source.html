189 string fullHelpMessage(void)
190  {
191  string s =
192 "\n"
193 " SYNOPSIS\n"
194 "\n"
195 " Compute lipid order parameters from C-H bond vectors\n"
196 "\n"
197 " DESCRIPTION\n"
198 "\n"
199 " This program computes order parameters analogous to those measured \n"
200 " via deuterium quadrupolar splitting on perdeuterated lipid chains.\n"
201 " It can correctly handle lipids specified using either 1 residue per lipid\n"
202 " (e.g. POPC) or 3 residues per lipid (e.g. PALM-PCGL-OLEO), although it \n"
203 " requires the presence of explicit hydrogens on the chains. In addition, \n"
204 " it can calculate the order parameters assuming the magnetic field is \n"
205 " oriented along the x- or y-axis, in addition to the more conventional z. \n"
206 " Finally, it has the ability to correctly estimate statistical errors in \n"
207 " the order parameters using block averaging applied to the full system.\n"
208 "\n"
209 " Formats for lipids\n"
210 "\n"
211 " The CHARMM27 force field typically represented lipids using 3 residues\n"
212 " (e.g. POPC was PALM-PCGL-OLEO), while the newer CHARMM36 force field\n"
213 " represents each lipid as a single residue. The code handles both\n"
214 " formats; the user has the option to explicitly tell it which format to \n"
215 " expect using the flags --1 and --3 (for 1 and 3 residues per lipid, \n"
216 " respectively). If neither flag is invoked, the code attempts to figure \n"
217 " it out based on the names of the carbons in the selection; if the \n"
218 " selection contains names like C3, its presumed to be 3/lipid format, \n"
219 " while the presence of names like C213 suggests 1/lipid format. The \n"
220 " code outputs which case it thinks it has found, so that the user can \n"
221 " override it if necessary.\n"
222 "\n"
223 "\n"
224 " Specifying which carbons to examine\n"
225 "\n"
226 " This is done in 2 parts: First, you give a selection on the command line\n"
227 " that says which chains you want to examine. Second, you say which carbon\n"
228 " positions you want to look at within that range (e.g. 2 - 16 for a \n"
229 " palmitoyl chain). The program figures out which atoms match, and \n"
230 " identifies the relevant hydrogen atoms. See the example section for more \n"
231 " details.\n"
232 "\n"
233 " Note: the file used to specify the system _must_ have connectivity \n"
234 " information (e.g. a psf file) for this program to work correctly. \n"
235 " If you're using gmxdump2pdb.pl to make a PSF/PDB from gromacs, you'll\n"
236 " need to run it with the --hydrogens option.\n"
237 "\n"
238 " Orientation of the magnetic field\n"
239 "\n"
240 " As a rule, order parameters are measured with the magnetic field \n"
241 " presumed to be oriented along the membrane normal. However, with \n"
242 " oriented samples one has the option to tilt the sample with respect \n"
243 " to the field. The present version of the code supports a subset of \n"
244 " this capability, allowing you to compute the order parameters as if\n"
245 " the field were oriented along the x or y axis, using the flags --x_axis\n"
246 " and --y_axis.\n"
247 "\n"
248 " Note: these results are indistinguishable experimentally, and should \n"
249 " be identical in the simulation up to statistical noise. If you're\n"
250 " interested in these results, it's probably best to compute both x- \n"
251 " and y-axis fields in order to verify that nothing funny is going on.\n"
252 "\n"
253 " Estimating statistical error using bootstrapping\n"
254 "\n"
255 " --block_average filename turn on block averaging, output \n"
256 " additional info to filename (see\n"
257 " below for explanation of the file\n"
258 " format)\n"
259 " --ba_first integer lower number of blocks to consider\n"
260 " as part of the plateau (col 2 in the\n"
261 " block_average file).\n"
262 " --ba_last integer upper number of blocks to consider\n"
263 " as part of the plateau (col 2 in the\n"
264 " block_average file).\n"
265 " --timeseries filename output a timeseries for the average \n"
266 " order parameter for each carbon position\n"
267 "\n"
268 " Estimating the statistical error in the order parameters computed from\n"
269 " molecular dynamics simulations is tricky, but it's not clear what \n"
270 " constitutes a single independent sample. Clearly, the orientation of \n"
271 " a given C-H bond is correlated in successive frames. However, there is\n"
272 " also the correlation between the 2 or 3 hydrogens bound to the same \n"
273 " carbon. More problematically, _all_ of the bonds are correlated to the\n"
274 " area per lipid for the system as a whole, which fluctuates slowly, \n"
275 " often on the timescale of tens of nanoseconds. So, while there may\n"
276 " be a total of 100 lipids in a system, that doesn't mean each frame has \n"
277 " anything like 200 (or even 100) independent measurements for the order\n"
278 " parameter at a particular carbon position. \n"
279 "\n"
280 " The approach taken here is use apply block averaging (Flyvbjerg and \n"
281 " Petersen, J Chem Phys, 1989, 91, 461-467), applied to the instantenous\n"
282 " value of the order parameter for a given carbon position, averaged over \n"
283 " the full system. In this method, the standard error in the mean from\n"
284 " a time series is estimated by computing the variance in the average \n"
285 " values computed over consecutive subsets of the time series; if the \n"
286 " value plateaus as the blocks get long, then that value is the standard\n"
287 " error. However, since the plateau is often noisy (since you don't have\n"
288 " as many of the long blocks), the code offers the option to average \n"
289 " over a range of block sizes. \n"
290 " \n"
291 " Since I can't know in advance what range of block sizes will produce a \n"
292 " plateau for any individual series, I let the user control this range. \n"
293 " I suggest running the program twice: run it with a guess for the \n"
294 " averaging range (or use the defaults). Then, plot the data in the file \n"
295 " containing the block average output and look at what range of block sizes\n"
296 " (if any) contains the plateau. This file is formatted as follows: the \n"
297 " first column specifies the carbon number, the second the block number,\n"
298 " the third the block size in frames, and the 4th the standard error \n"
299 " estimate; plot the 4th vs the 3rd to get the variation in the standard\n"
300 " error with block size (plot \"file\" using 3:4 w lp in gnuplot, for \n"
301 " example), and then read off from the file which blocks are in that range.\n"
302 "\n"
303 " If you wish to do your own analysis of the variation of the average \n"
304 " order parameter, you can specify the --timeseries flag, which will dump\n"
305 " out the average value for each carbon position at each time point, with\n"
306 " each carbon in its own column.\n"
307 " \n"
308 " \n"
309 " Notes for use with Gromacs:\n"
310 "\n"
311 " If you want to use order_params to analyze data generated with gromacs, \n"
312 " there are two caveats you must be aware of. First, this program assumes\n"
313 " you have explicit hydrogens, and will not work if you use a lipid model\n"
314 " without them; if you used implicit hydrogens, you'll have to use the \n"
315 " builtin gromacs tool instead. Second, as mentioned above you'll need\n"
316 " to use a model file that has explicit bonds in it, INCLUDING BONDS TO\n"
317 " HYDROGEN. The most common usage of gromacs will have this these bonds\n"
318 " replaced by constraints, so when you use gmxdump2pdb.pl to make your\n"
319 " fake psf file, be sure to give the \"--hydrogens\" option so that the\n"
320 " bonds to hydrogen are generated correctly.\n"
321 "\n"
322 " EXAMPLE\n"
323 "\n"
324 " order_params namd.psf merged_1ns.dcd '(segid =~ \"L([0-9]+)\") && resname == \"PALM\"' 2 16 --block_average foo.dat --ba_first 2 --ba_last 5\n"
325 "\n"
326 " This is a moderately complicated example: namd.psf specifies the system,\n"
327 " while merged_1ns.dcd is the trajectory. The next argument is a selection\n"
328 " that specifies the palmitate chains for lipids with segment names L and a\n"
329 " number; this assumes that the lipids are in the old C27 \n"
330 " 3-residues-per-lipid format). The values 2 and 16 specify that we want\n"
331 " to look at carbons 2 to 16 within the chain (all of them, in this case).\n"
332 " The rest of the command line turns on block averaging (sending the \n"
333 " output to foo.dat), and specifies that the plateau region should be\n"
334 " estimated as the range between the 2 block data point and the 5 block\n"
335 " data point. \n"
336 "\n"
337 " If you're using 1-residue-per-lipid format (as in the new C36 force \n"
338 " field), the selection string would look a bit more complicated. For\n"
339 " example, to select the docosahexaenoyl chain of an SDPE lipid, the string would\n"
340 " be \n"
341 " 'resname == \"SDPE\" && name =~ \"^C2[1-9][0-9]?\"'\n"
342 " This selects from residues named \"SDPE\" those atoms with names beginning\n"
343 " with C2, followed by a number from 1-9, and perhaps a second integer.\n"
344 "\n"
345 " The resulting output will look something like:\n"
346 " # order_params 'namd.psf' 'npgt_bgw.2.dcd' 'resname == \"SDPE\" && name =~ \"^C2[1-9][0-9]?\"' '2' '22' '--block_average' 'foo.dat' - alan (Thu Mar 15 09:23:44 2012) {/home/nleioatts/Public/c36} [2.0.0 120313]\n"
347 " # guessing there is 1 residue per lipid\n"
348 " # Carbon S_cd +/- BSE\n"
349 " 2 0.01811 0.01311 0.00164\n"
350 " 3 0.05558 0.02481 0.00787\n"
351 " 4 0.08293 0.03523 0.01002\n"
352 " 5 0.08138 0.03229 0.00512\n"
353 " 6 0.01762 0.02208 0.00303\n"
354 " (with more lines)\n"
355 "\n"
356 " The first column is the carbon number, the second is the order parameter,\n"
357 " the third is the stdev of the value, and the last column is the \n"
358 " estimate of the statistical error from block averaging. If the command\n"
359 " line hadn't included --block_average, the last column would be absent.\n"
360  ;
361  return(s);

381 skip = tropts->skip;
382 system_filename = tropts->model_name;
383 traj_filename = tropts->traj_name;
384 vector<uint> framelist = tropts->frameList();
385

387 // (just copied from the BasicTrajectory object)

391 
392 
393 cout << "# " << invocationHeader(argc, argv) << endl;
394 // TODO: add a check for connectivity
395 
396 // NOTE: We assume the selection is a list of all of the relevant carbon atoms.
397 // We'll split it into the individual carbon positions ourselves, then
398 // figure out the relevant hydrogens from connectivity.
399 AtomicGroup main_selection = selectAtoms(system, selection);
400 
401 // Do we need to figure out how many residues per lipid?
402 // If both are false, then neither option was set on the command line
403 if ( !one_res_lipid && !three_res_lipid)
404  {
405  // Number of residues per lipid wasn't set, try to figure it out.
406  // If we find things that look like "C3" (one number after the carbon),
407  // it has to be 3 residues per lipid. If we find things that look like
408  // "C213" (3 digits after the C), it has to be 1. If we find neither,
409  // it could be because the user just selected a subset of carbon positions
410  // to look at; we'll guess 3-residues per and inform the user.
411  // If we find both, we're screwed, and will die screaming.
412  // Note: this gets CHARMM (pre- and post-C36 parameters) right, but hasn't been
413  // tested on anything else. GROMACS doesn't have hydrogens, so this whole code
414  // won't work, and I don't have access to anything else (eg AMBER GAFF).
415  string sel1 = string("name =~ \"^C[1-9]$\"");
416  string sel3 = string("name =~ \"^C[1-9][0-9][0-9]$\"");
417  AtomicGroup a1 = selectAtoms(main_selection, sel1.c_str());
418  AtomicGroup a3 = selectAtoms(main_selection, sel3.c_str());
419 
420  if ( (a1.size() > 0) && (a3.size() == 0) )
421  {
422  // found only 1 digit numbers and no 3 digit numbers,
423  // so it must be 3 residues per lipid
424  three_res_lipid = true;
425  cout << "# guessing there are 3 residues per lipid"
426  << endl;
427  }
428  else if ( (a3.size() > 0) && (a1.size() == 0) )
429  {
430  // found only 3 digit numbers and no 1 digit ones, so
431  // it must be 1 residue per lipid
432  one_res_lipid = true;
433  cout << "# guessing there is 1 residue per lipid"
434  << endl;
435  }
436  else if ( (a1.size() > 0) && (a3.size() > 0) )
437  {
438  // found both 1 and 3 digit numbers, which is very weird.
439  // Notify the user and tell them to pick manually.
440  cerr << "Couldn't figure out whether you have 1 or 3 residues "
441  << "per lipid molecules."
442  << endl;
443  cerr << "You'll need to specify this manually. Exiting...."
444  << endl;
445  exit(-1);
446  }
447  else if ( (a1.size() == 0) && (a3.size() == 0) )
448  {
449  // we didn't find 1 or 3 digit numbers. This can happen if the
450  // user only looks at a subset of carbons -- eg carbons 5-7 in one
451  // residue format, or carbons 12-15 in 3 residue format.
452  cerr << "Can't unambiguously tell whether you've got 1 or 3 "
453  << "residues per lipid. I'm guessing 3, "
454  << endl;
455  cerr << "but if this guess is wrong, you'll need to rerun with "
456  << "the correct value specified."
457  << endl;
458  three_res_lipid = true;
459  }
460  }
461 
462 // At this point, we must have either three_res_lipid or one_res_lipid set.
463 // This means we only have to test one to decide what to do.
464 assert(one_res_lipid != three_res_lipid);
465 
466 // Now break into individual carbons
467 vector<AtomicGroup> selections;
468 for (int i =first_carbon; i<=last_carbon; i++)
469  {
470  char carbon_name[4];
471  string sel_string = selection;
472  sprintf(carbon_name, "%d", i);
473  string name;
474  if (three_res_lipid)
475  {
476  name = string(" && name == \"C");
477  }
478  else // implies one_res_lipid is true
479  {
480  name = string(" && name =~ \"C[123]");
481  }
482  name += string(carbon_name) + string("\"");
483  sel_string.insert(sel_string.size(), name);
484  selections.push_back(selectAtoms(main_selection, sel_string.c_str()));
485  }
486 
487 // Now, figure out which hydrogens go with each carbon selected
488 // hydrogen_list mimics the structure of selections, so the hydrogens
489 // bound to the jth carbon of the ith selection will be found at
490 // hydrogen_list[i][j].
491 vector< vector<AtomicGroup> > hydrogen_list(selections.size());
492 HydrogenSelector hyd_sel;
493 for (unsigned int i=0; i<selections.size(); i++)
494  {
495  AtomicGroup *s = &(selections[i]);
496  AtomicGroup::Iterator iter(*s);
497  pAtom p;
498  while ((p = iter()))
499  {
500  vector<int> atom_ids = p->getBonds();
501  AtomicGroup bonded = system.groupFromID(atom_ids);
502  AtomicGroup bonded_hydrogens = bonded.select(hyd_sel);
503  if (bonded_hydrogens.size() == 0)
504  {
505  cerr << "No hydrogens found bound to carbon atom: "
506  << p->name()
507  << endl;
508  cerr << "This could happen if your lipid doesn't have explicit\n "
509  << "hydrogens, if you gave an incorrect selection string,\n "
510  << "or if the model file you supplied doesn't have "
511  << "connectivity information.";
512  cerr << endl;
513  cerr << "If you're using a PSF generated by gmxdump2pdb.pl, you\n"
514  << "should rerun it using the --hydrogens option so that \n"
515  << "bonds to hydrogen are correctly represented.\n";
516  cerr << endl;
517  exit(-1);
518  }
519 
520  hydrogen_list[i].push_back(bonded_hydrogens);
521  }
522  }
523 
524 #ifdef DEBUG
525 // Check to see if the correct hydrogens were found
526 for (unsigned int i=0; i<selections.size(); i++)
527  {
528  AtomicGroup *g = &(selections[i]);
529  cerr << "total atoms in sel " << i << "= " << g->size() << endl;
530  for (int j=0; j<g->size(); j++)
531  {
532  pAtom carbon = g->getAtom(j);
533  // get the relevant hydrogens
534  AtomicGroup *hyds = &(hydrogen_list[i][j]);
535  cerr << *carbon << endl;
536  cerr << *hyds << endl;
537  }
538  }
539 #endif

544 
545 // We're going to accumulate the time series of average values for each
546 // carbon position, and turn this into an average at the end.
547 // This will let us do better uncertainty analysis.
548 vector<vector<float> > values;
549 vector<int> counts;
550 values.resize(selections.size());
551 for (uint i=0; i<values.size(); i++)
552  {
553  values[i].insert(values[i].begin(), num_frames, 0.0);
554  }
555 counts.insert(counts.begin(), selections.size(), 0);

561  if (!timeseries_outfile.good())

566  << "Proceeding without outputing time series"
567  << endl;
568  dump_timeseries = false;
569  }

572  // write the header

575  {
576  timeseries_outfile << i << "\t";
577  }
578  timeseries_outfile << endl;
579  }

581  }

583 // loop over frames in the trajectory
584 int frame_index = 0;
585 
586 for (uint i=0; i<framelist.size(); ++i)
587  {
588  traj->readFrame(framelist[i]);
589  traj->updateGroupCoords(system);
590 
591
